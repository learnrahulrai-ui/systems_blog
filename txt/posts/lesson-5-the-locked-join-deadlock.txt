  SPECIAL -- LESSON 5: THE LOCKED JOIN DEADLOCK
  Lesson 5: The Locked Join Deadlock
  ============================================================================================



  Fix one crash and get a worse failure in return: not a crash this time, a program that never
  ends, consuming 0% CPU, forever. The fix from lesson 4 is in place, but the join call sits
  inside the same scope as a locked mutex.

```text
~BadLogger() {
    std::unique_lock<std::mutex> lock(mtx_);
    stop_ = true;
    cv_.notify_one();
    
    if (background_writer_.joinable()) {
        background_writer_.join(); // still holding the lock here
    }
} // lock drops only now
```

  experiment/todo1_locked_join_bug.cpp isolates exactly what happens mechanically when this
  runs. Compile it yourself, no Makefile, no CMake:
```text
g++ -std=c++17 -o demo lesson-5-locked-join-deadlock.cpp
./demo
```

  --------
  THE RUN FREEZES, IT DOES NOT CRASH
  Running the experiment prints these lines and then stops:
```text
Starting program...
[Background Thread] Started, waiting for work...
[Main Thread] Ending scope, ~BadLogger will be called.
[Main Thread] Destructing BadLogger... (Acquiring lock)
[Main Thread] Setting stop_ = true
[Main Thread] Calling cv_.notify_one()
[Main Thread] Calling background_writer_.join() while STILL holding the lock... (DEADLOCK!)
```
  The program does not abort. It consumes 0% CPU and never returns. A deadlock, not a crash --
  the process has to be killed by hand.

  --------
  SO WHY DOES NOTIFYING THE THREAD NOT WAKE IT
  A condition variable has a specific wakeup sequence. While the background thread sleeps on
  cv_.wait, it has internally dropped the mutex -- that is the whole point of wait, so the
  main thread can get the lock to make its change.

  When the main thread calls cv_.notify_one, the background thread wakes up. But before wait
  is allowed to actually return, it must re-acquire the mutex first.

  And the join call sits inside the scope of the same lock. Which produces a standoff:
  1. The background thread is stuck trying to re-acquire the mutex so wait can finish waking
     up.
  2. The main thread is holding that same mutex and is stuck inside join, waiting for the
     background thread to finish.

  Each one is waiting on the other, forever.

  --------
  TRACE IT DOWN TO THE TWO CALLS THAT NEVER RETURN
  At the function level, exactly two calls freeze halfway and never return:

  1. thread::join, called inside the destructor. The main thread enters this call and never
     comes back -- it is permanently asleep, waiting for the background thread to exit.
  2. condition_variable::wait, called inside processQueue. The background thread enters this
     to sleep. Notified, it tries to wake and return, but cannot grab the lock, so it gets
     stuck inside wait's own internal locking. It never returns from wait.

  Because wait never finishes, processQueue never finishes. Because processQueue never
  finishes, join never finishes. Because join never finishes, the destructor never finishes.
  One stuck call blocks the entire chain above it.

  --------
  SO THE LOCK HAS TO DROP BEFORE THE JOIN ON ANOTHER THREAD BEGINS
```text
~BadLogger() {
    {
        std::unique_lock<std::mutex> lock(mtx_);
        stop_ = true;
    } // Lock drops here!
    
    cv_.notify_one();
    if (background_writer_.joinable()) {
        background_writer_.join();
    }
}
```
  Now when the background thread wakes up the lock is free. It re-acquires it, reads
  stop_ == true, breaks its loop, and returns from processQueue -- which is what finally lets
  join, and the destructor, succeed.

  >> STANDARD NAMES
     each side waiting on the other forever -> deadlock
     wait's requirement before it can return -> re-acquiring the mutex
     dropping a lock before a blocking call  -> scoping the lock to end before join/wait
