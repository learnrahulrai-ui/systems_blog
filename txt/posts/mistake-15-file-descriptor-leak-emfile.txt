  SPECIAL -- IPC MISTAKE: FILE DESCRIPTOR LEAK EMFILE
  Mistake 15: File Descriptor Leak EMFILE
  ============================================================================================


  A file descriptor is just a small integer -- 4, 5, 6, climbing by one each time accept()
  succeeds. Forget to close() it after handling a client, and that integer is never returned
  to the pool. Nothing crashes on client 1, or client 100. It crashes on whichever client
  happens to be the one that exhausts the process's entire descriptor table.

  --------
  FIRST, HOW THE LEAK ACCUMULATES
```text
accept() client ---> returns fd N ---> handle client ---> (no close(fd)) ---> leaks fd N
repeat 2048 times ---> hits the limit ---> accept() returns -1 with EMFILE ---> server breaks
```

  --------
  THEN THE ACTUAL CODE THIS RUNS ON -- COMPILE IT AND RUN IT YOURSELF

  No Makefile, no CMake, one command:
```text
g++ -std=c++17 -o demo mistake-15-file-descriptor-leak-emfile.cpp
./demo support     -- terminal 1, accepts forever, NEVER closes client_fd
./demo consumer    -- terminal 2, fires 5000 clean connect+close cycles
```

```cpp
#include <stdio.h>
#include <iostream>
#include <cstring>
#include <unistd.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <arpa/inet.h>

void support() {
    int server_fd = socket(AF_INET, SOCK_STREAM, 0);
    int opt = 1;
    setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, (const char*)&opt, sizeof(opt));

    struct sockaddr_in address;
    address.sin_family = AF_INET;
    address.sin_addr.s_addr = INADDR_ANY;
    address.sin_port = htons(8080);
    bind(server_fd, (struct sockaddr*)&address, sizeof(address));

    listen(server_fd, 2048); // Massive backlog so the OS doesn't reject them

    std::cout << "Support started. Accepting connections nonstop!" << std::endl;

    while (true) {
        int client_fd = accept(server_fd, nullptr, nullptr);

        if (client_fd == -1) {
            std::cout << "\nBOOM! The OS refused to give us another file descriptor!" << std::endl;
            perror("accept failed");
            break; // End the loop
        }

        std::cout << "Accepted client on fd: " << client_fd << std::endl;

        // WE ARE NOT CLOSING IT! WE ARE LEAKING EVERY SINGLE FD!
        // close(client_fd);
    }
    close(server_fd);
}

void consumer() {
    std::cout << "Consumer launching a massive connection flood..." << std::endl;

    for (int i = 0; i < 5000; i++) {
        int client_fd = socket(AF_INET, SOCK_STREAM, 0);

        if (client_fd == -1) {
            std::cout << "\nBOOM! Consumer ran out of file descriptors!" << std::endl;
            perror("socket failed");
            break;
        }

        struct sockaddr_in address;
        address.sin_family = AF_INET;
        address.sin_port = htons(8080);
        inet_pton(AF_INET, "127.0.0.1", &address.sin_addr);

        if (connect(client_fd, (struct sockaddr *)&address, sizeof(address)) == -1) {
            perror("connect failed");
            break;
        }

        // THIS CONSUMER IS PERFECTLY CLEAN!
        // It does its job and properly closes its socket.
        close(client_fd);
    }
}

int main(int argc, char* argv[]) {
    if (argc < 2) { printf("Usage: ./demo <support|consumer>\n"); return 0; }
    if (strcmp(argv[1], "support") == 0) support();
    if (strcmp(argv[1], "consumer") == 0) consumer();
    return 0;
}
```

  Running this prints:

      Usage: ./demo <support|consumer>

  --------
  SO WHAT THE REAL COMMIT SAYS ABOUT THIS
  commit 9e354c2654cdff1edf15f0ef144a59cf5c94f48d, "The CLOSE_WAIT Zombie Apocalypse (EMFILE
  bug)". A stress test fired 5000 perfectly clean consumers at the server -- each one
  connecting, receiving its data, and calling close() correctly on its own end. The server
  still leaked, because a clean consumer disconnect does not force the server side to release
  anything on its own. The server accepted exactly 2047 connections (fd 4 through 2050) before
  crashing on the 2048th.

  Why 2048 specifically: the default Microsoft C runtime enforces a hard cap of exactly 2048
  low-level I/O file descriptors per process. The 2048th connection tripped WSAENOBUFS, and
  accept() returned -1.

  --------
  SO WHY DOES THE CONSUMER'S OWN CLEANLINESS NOT SAVE THE SERVER
  Closing a socket is a two-sided act. The consumer closing its end sends a FIN and lets go of
  its own descriptor -- but that only tells the server the peer is gone. It does not force the
  server to release its side. Left unclosed, the server's socket sits in CLOSE_WAIT,
  permanently holding one descriptor slot, no matter how correctly the other side behaved.

  A file descriptor is process-local memory, not shared state -- it frees instantly on close(),
  but only on the close() call that belongs to that process. One missing close(client_fd) on
  the server is a slow-burning fuse: invisible for the first 2047 connections, fatal on the
  2048th.

  >> STANDARD NAMES
     socket held open after the peer's clean close -> CLOSE_WAIT
     the OS running out of descriptor slots         -> EMFILE (2048 per process, MSVC CRT default)
