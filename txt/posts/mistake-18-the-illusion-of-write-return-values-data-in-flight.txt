  SPECIAL -- IPC MISTAKE: THE ILLUSION OF WRITE RETURN VALUES DATA IN FLIGHT
  Mistake 18: The Illusion of write Return Values Data in Flight
  ============================================================================================


  When `write(100MB)` returns `638,976`, it DOES NOT mean the consumer received that data. It only means the local OS Kernel *accepted* 638KB into its Send Buffer. If the consumer suddenly aborts (RST), that data is permanently lost mid-flight, even though `write()` reported it as "written".

  --------
  DATAFLOW AND PROCESS CONSTRAINTS

```text
write(100MB) ---> returns 638,976 (bytes accepted into local OS Kernel Send Buffer)
                           |
              Does NOT mean consumer received it!
                           |
       If consumer aborts, data in flight is vaporized!
```

  When benchmarking a server's push capacity across 5 different, brand-new processes connecting to 5 **different** IP addresses (`127.0.0.1` through `127.0.0.5`), the bytes successfully pushed into the non-blocking socket did not reset. Instead, they scaled geometrically:

  Even when rewriting the experiment to use zero threads, zero shared variables, and zero loops (a completely flat script), the results remained identical.

  --------
  VIRTUALLOCK PROOF OF MEMORY LOCKING
  To definitively prove that user-space memory allocation has no effect on the TCP window, we rewrote the C++ program to use `VirtualAlloc` and `VirtualLock`, forcing the OS to lock the 100MB user-space buffer into physical, non-pageable RAM.

  The Kernel ignored the user-space memory architecture completely and applied the exact same temporal route caching heuristics, jumping across strict mathematical "gears" based on a `212,992`-byte interval:

```text
Run 1 (127.0.0.1): 1,064,960 bytes (Baseline)
Run 2 (127.0.0.2): 2,768,896 bytes (+ 8x 212,992)
Run 3 (127.0.0.3): 1,490,944 bytes (+ 2x 212,992)
Run 4 (127.0.0.4): 1,277,952 bytes (+ 1x 212,992)
Run 5 (127.0.0.5): 2,981,888 bytes (+ 9x 212,992)
```

  --------
  WHAT CHANGED IN OUR RUNTIME ENVIRONMENT

  Nothing in the user-space code changed. The operating system kernel intervened.

  --------
  ROOT CAUSE: RFC 2140 AND INTERFACE ROUTE CACHING

  When a C++ program calls `write()` or `send()`, the program does not touch the network. It passes a buffer to the OS Kernel (the `tcpip.sys` driver on Windows), which copies the data into the **Non-Paged Pool**.

  The OS Kernel is a persistent, global entity. It tracks the speed and health of its physical (and virtual) network interfaces to optimize throughput.

  According to **RFC 2140 (TCP Control Block Interdependence)**, modern OS networking stacks deliberately cache TCP Congestion Window (`cwnd`) and Slow Start Threshold (`ssthresh`) metrics. Crucially, Windows caches these metrics against the **Subnet Route** or **Network Interface**, not just the individual IP.

  Because `127.0.0.1`, `127.0.0.2`, and `127.0.0.3` all belong to the `127.0.0.0/8` subnet, they all route through the exact same Virtual Loopback Interface.

  When Run 1 proved the loopback interface was fast, the global Kernel updated its heuristic cache for the entire interface. When Run 2 (a brand new C++ process) connected to `127.0.0.2`, the Kernel instantly applied the "warmed up" metrics from Run 1, granting it a massive TCP window. When the consumer application slept for 13 seconds (proving to be a "slow reader"), the Kernel punished the interface, mathematically shrinking the window back down by Runs 4 and 5 to prevent buffer bloat.

  --------
  WE BIND OUR THREADS TO REALITY

  Your C++ program is entirely at the mercy of global Kernel heuristics. Relying on blocking sockets or implicit kernel buffering makes your application's performance unpredictable, as it will inherit the network state left behind by entirely unrelated processes.

  Asynchronous, non-blocking I/O (`poll`, `epoll`, `IOCP`) is the only way to decouple your thread execution from the Kernel's unpredictable TCP scaling algorithms.
