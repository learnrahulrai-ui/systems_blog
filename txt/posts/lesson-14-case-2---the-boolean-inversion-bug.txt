  SPECIAL -- LESSON 14: CASE 2 - THE BOOLEAN INVERSION BUG
  Lesson 14: Case 2 - The Boolean Inversion Bug
  ============================================================================================



  Case 2 from the permutation matrix: the queue check is dropped from the wakeup lambda,
  leaving only the shutdown check.
```text
cv_.wait(lock, [this]() { return stop_; });
```
  Two real questions came up while tracing this one, plus one wrong mental model that explains
  the whole bug once it is corrected.

  --------
  FIRST: IS TAKING A LOCK IN LOG() ACTUALLY BAD
  No -- it is mandatory. A web server can have 100 users hit the API in the same millisecond,
  which means 100 HTTP threads calling log() at once. std::queue is not thread-safe: two
  threads pushing to it without a lock will corrupt its internal pointers and segfault the
  server. The lock is what guarantees only one thread pushes at a time.

  --------
  SECOND: DOES PROCESSQUEUE NEED TO NOTIFY ANYONE
  No. processQueue is the consumer -- it eats the data. main() does not wait to be told when a
  log is written; it drops the log off and moves on. Since nothing is waiting on processQueue,
  processQueue has nothing to notify.

  --------
  THIRD, AND THIS IS THE ACTUAL BUG: WHAT DOES FALSE MEAN HERE
  The mental model that broke: treating cv_.wait like a bouncer checking IDs, where false means
  "you're fine, go ahead."

  It is the opposite. cv_.wait is a tollbooth waiting for a green light. If the lambda returns
  false, the thread is told the light is still red -- go back to sleep. Only true lets it pass.

  Trace the logic with that corrected rule:
  1. main() pushes a log and calls notify_one().
  2. The background thread wakes up.
  3. It evaluates return stop_.
  4. stop_ is false.
  5. False means red light. The thread goes straight back to sleep.
  6. It never reaches log_queue_.pop().

  --------
  SO THIS IS THE SAME FAILURE AS LESSON 7, FROM THE OPPOSITE DIRECTION
  Because the thread keeps going back to sleep, logs pile up in RAM without limit. Not a single
  log reaches disk while the server is running -- only when main() finally shuts the server
  down does stop_ become true and the thread wake for real. If the server crashes before that
  point, everything queued is lost.

  >> STANDARD NAMES
     wait() returning false        -> stay asleep, not "proceed"
     wait() returning true         -> the only way past the predicate
     this exact failure by name    -> flush-on-shutdown (lesson 7)
