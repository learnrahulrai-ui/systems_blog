  SPECIAL -- LESSON 14: LESSON 14: CASE 2 - THE BOOLEAN INVERSION BUG
  Lesson 14: Lesson 14: Case 2 - The Boolean Inversion Bug
  ============================================================================================



  --------
  THE SETUP
  We removed the queue check from the lambda, leaving only the shutdown check:
```text
cv_.wait(lock, [this]() { return stop_; });
```

  --------
  YOUR BRAIN DUMP
  You asked two critical questions and made one massive error in logic.

  --------
  1. "WE TRY TO TAKE A LOCK IN LOG WHY ;; TAKING A LOCK IS BAD"
  Taking a lock in `log()` is **NOT BAD**. It is absolutely mandatory.
  You are building a web server. 100 users might hit your API in the exact same millisecond. That means 100 HTTP threads will call `Logger::log()` at the exact same time.
  `std::queue` is **not thread-safe**. If two threads try to push to it without locking, the internal memory pointers will instantly corrupt and your server will segfault. The lock ensures only one thread pushes at a time.

  --------
  2. "DO WE NOTIFY IN PROCESS QUEUE AT ALL DO WE NEED TO ??"
  **No.** `processQueue` is the consumer. It eats the data. `main()` does not care when the data is written; it just drops it off and walks away. Because no one is waiting on `processQueue`, `processQueue` does not need to notify anyone.

  --------
  3. THE FATAL FLAW: "SEE STOP IS FALSE ;; OK NO PORIBLEM WE GO AHEAD"
  This is the core of your misunderstanding!

  You think `cv_.wait` acts like a bouncer checking for fake IDs (where false = good to go).
  **FALSE.** `cv_.wait` acts like a tollbooth waiting for a Green Light!

  If the lambda returns `FALSE`, the thread says: *"Nope, the light is still red, the condition has not been met. Go back to sleep!"*
  If the lambda returns `TRUE`, the thread says: *"Green light! I am allowed to pass!"*

  Let's trace your logic with this corrected rule:
  1. `main()` pushes a log and calls `notify_one()`.
  2. The Background Thread wakes up.
  3. It evaluates `[this]() { return stop_; }`.
  4. `stop_` is **FALSE**.
  5. The Background Thread sees FALSE. It says *"Red Light!"* and **goes permanently back to sleep**.
  6. It NEVER reaches `log_queue_.pop()`.

  --------
  THE RESULT
  This is the **Flush-on-Shutdown Bug** we covered in Lesson 7!
  Because the thread goes back to sleep, the logs just pile up in RAM. It will never write a single log to disk while the server is running. It will only wake up when `main()` shuts down the server (and `stop_` finally becomes TRUE). If the server crashes before then, you lose everything.
