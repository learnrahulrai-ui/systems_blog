  SPECIAL -- LESSON 12: THE PERMUTATION MATRIX
  Lesson 12: The Permutation Matrix
  ============================================================================================



  The Asynchronous Logger relies on exactly 4 variables spread across two lines of code to function correctly:
```text
// Line A: The Tollbooth
cv_.wait(lock, [this]() { return stop_ || !log_queue_.empty(); });

// Line B: The Exit Door
if (stop_ && log_queue_.empty()) break;
```

  If you remove **any single variable** from these two lines, the server will suffer a catastrophic failure. Let's phase through the debugger trace for all 5 of your permutations.

  ---

  --------
  CASE 1: REMOVE STOP_ FROM THE LAMBDA
  `cv_.wait(lock, [this]() { return !log_queue_.empty(); });`

  **The Bug: The Shutdown Deadlock**
  1. The server runs fine for 100 days.
  2. `main()` initiates shutdown. `Logger::~Logger()` sets `stop_ = true` and calls `notify_one()`.
  3. The queue is empty.
  4. The Background Thread wakes up and evaluates `!log_queue_.empty()`. It evaluates to `false`.
  5. The Background Thread immediately goes back to sleep. It never reaches the `if(stop_) break;` door.
  6. The Main Thread is permanently stuck in `join()`. **Deadlock.**

  ---

  --------
  CASE 2: REMOVE !LOG_QUEUE_.EMPTY() FROM THE LAMBDA
  `cv_.wait(lock, [this]() { return stop_; });`

  **The Bug: Flush-on-Shutdown / Power Outage Data Loss**
  (We covered this in Lessons 7 & 8)
  1. The server runs for 100 days.
  2. The Background Thread refuses to wake up because `stop_` is false.
  3. The RAM fills up with gigabytes of unwritten logs.
  4. If the server loses power, all 100 days of logs are instantly destroyed because they were never written to disk.

  ---

  --------
  CASE 3: REMOVE STOP_ FROM THE IF STATEMENT
  `if (log_queue_.empty()) break;`

  **The Bug: Premature Thread Death**
  1. The server boots up. `main()` logs "User logged in".
  2. The Background Thread wakes up, pops the message, and writes it to disk.
  3. The queue is now empty.
  4. The execution moves downward and hits `if (log_queue_.empty()) break;`.
  5. Because the queue is empty, the thread instantly `break`s and **commits suicide**.
  6. For the next 100 days of the server's life, there is no Background Thread. All future logs pile up in memory forever.

  ---

  --------
  CASE 4: REMOVE LOG_QUEUE_.EMPTY() FROM THE IF STATEMENT
  `if (stop_) break;`

  **The Bug: Truncated Shutdown (Data Loss)**
  (We covered this in Lesson 11)
  1. `main()` initiates shutdown while there are still 500 unwritten logs in the queue.
  2. The Background Thread wakes up. It hits `if (stop_) break;`.
  3. It instantly breaks and dies.
  4. The 500 logs are abandoned in memory and destroyed.

  ---

  --------
  CASE 5: REMOVE THE ENTIRE IF STATEMENT
  (No break statement at all)

  **The Bug: The "No Exit" Segfault**
  1. `main()` initiates shutdown.
  2. `Logger::~Logger()` sets `stop_ = true` and `notify_one()`.
  3. The queue is empty.
  4. The Background Thread wakes up. The lambda `return stop_` is true, so it passes the tollbooth!
  5. Because there is no `break` statement to catch it, it moves downward to the draining logic:
```text
std::string msg = log_queue_.front();
```
  6. **BOOM.** Calling `.front()` on an empty queue instantly crashes the server with a Segmentation Fault (Exit Code 1).

  ---

  --------
  CONCLUSION
  Every single boolean check in those two lines is load-bearing.
  * The Lambda (`Line A`) controls **WHEN** the thread is allowed to wake up.
  * The If Statement (`Line B`) controls **WHEN** the thread is allowed to die.
