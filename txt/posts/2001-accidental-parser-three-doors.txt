  SPECIAL -- 2001: THREE DOORS INTO A PARSER THAT WAS ONLY ACCIDENTALLY CORRECT
  2001: Three Doors Into a Parser That Was Only Accidentally Correct
  ============================================================================================


  Part of the year-by-year series: [2000](2000-y2k-collision-three-doors.txt) came before
  this; [2002](2002-stale-index-three-doors.txt) comes next; the whole arc is in
  [The Decade in a Flat File](the-decade-in-a-flat-file.txt).

  --------
  BEFORE ANY DOOR: ONE FUNCTION, NOTHING ELSE ASSUMED
  This post needs one C++ fact. A vector keeps its elements in the order they were inserted,
  and back() returns the most recently inserted one -- not the biggest, not the one matching
  any ID, just whatever arrived last. Everything below is the story of a program betting its
  correctness on "whatever arrived last" and "the one I actually mean" being the same thing.

  --------
  FIRST, THE THREE DOORS
```text
door 1: CODEBASE     main() -> load_from_disk() at boot, save_to_disk() at shutdown
door 2: HAPPY user    boots, sends messages all day, shuts down cleanly, reboots -- history intact
door 3: ANGRY user    none, yet -- this year has no live writer at all. That is the whole point.
```
  There is no door 3 this year in the usual sense. That absence is exactly what makes next
  year's bug possible -- a correctness property that was never actually tested against the
  case that will eventually break it.

  --------
  THE SAVE, FROM DOOR 1
```text
for (size_t i = 0; i < all_conversations.size(); i++) {
    outfile << "CONV " << all_conversations[i].id << "\n";
    for each message: outfile << "MSG " << ... << "\n";
}
```
  One pass through RAM, in order, writing every conversation as one unbroken block before
  moving to the next.

  --------
  ? THE HAPPY PATH, FROM SCRATCH
  Door 2: the file this writes is read back in with one single assumption baked into the
  loader:
```text
all_conversations.back().messages.push_back(m);
```
  back() does not search for a conversation by ID. What guarantee does the code actually need
  to be holding, at the moment this line runs, for "whatever was pushed last" to be the correct
  answer? Name the guarantee precisely -- not "it usually works," the actual invariant.

  --------
  THE ANSWER, TRACED
  The guarantee is that physical position on disk and logical grouping are the same thing --
  every message immediately following a CONV line, with nothing else interleaved, belongs to
  that CONV. That guarantee holds here because exactly one writer, in one single pass, produced
  this file from a fully-formed, already-correct copy of the same data in RAM. The file is not
  independently correct. It is correct because the thing that wrote it was already correct and
  wrote in order. back() only works because nothing has ever been appended to this file outside
  of that one orderly save.

  --------
  ?? THE UNHAPPY PATH, FROM SCRATCH -- NOT A BUG IN THIS YEAR'S CODE
  Nothing in 2001 crashes, misroutes, or loses data. Before looking at any later year: what is
  the one operational change -- not a code change, an operational one -- that would break the
  back() guarantee without touching a single line of the parser? Name the change, then name
  what specifically about it breaks the invariant you just traced.

  --------
  THE ANSWER, TRACED
  The moment anything writes to this file outside of one single, complete, in-order save --
  most obviously, writing messages to disk live as they arrive instead of waiting for shutdown
  -- physical position on disk stops matching logical grouping. A message appended at 1:00 PM
  from one conversation and a message appended at 1:00:01 PM from a different one now sit next
  to each other on disk for a reason that has nothing to do with which conversation they belong
  to: whoever happened to send at that instant. back() was never wrong. The file it assumed
  never changes shape stopped existing the moment a second kind of writer showed up.

  --------
  SO THE 2001 CHECKPOINT
  The parser was not fragile because it was badly written. It was fragile because it encoded an
  assumption -- static snapshot, one writer, one pass -- as an unstated fact about the file
  rather than a checked one. The fix, when it eventually lands, is not a smarter parser. It is
  removing the assumption: making every message state its own conversation ID instead of
  inheriting one from whatever sits above it on disk. That fix does not belong to this year.
  This year's job is just naming, precisely, the invariant that is about to break.

  --------
  SEAM. The reasoning ends here; below, "accidentally correct" checked in C++.

  The claim is strong: the SAME parser, unmodified, produces correct grouping on a snapshot
  file and wrong grouping on an interleaved one. If that is true, the bug was never in the
  parser's code -- it was in the file's shape. One function, two hardcoded files, run both:

```cpp
#include <cstdio>
#include <string>
#include <vector>
#include <sstream>

struct Conversation {
    int id;
    std::vector<std::string> senders;
};

void parse_with_back(const char* label, const char* file_text) {
    std::vector<Conversation> all;
    std::istringstream file_stream(file_text);
    std::string type;
    while (file_stream >> type) {
        if (type == "CONV") {
            Conversation c;
            file_stream >> c.id;
            all.push_back(c);
        } else if (type == "MSG") {
            std::string sender, text;
            file_stream >> sender;
            std::getline(file_stream, text);
            all.back().senders.push_back(sender);
        }
    }
    printf("%s:\n", label);
    for (size_t i = 0; i < all.size(); i++) {
        printf("  CONV %d gets:", all[i].id);
        for (size_t j = 0; j < all[i].senders.size(); j++) {
            printf(" %s", all[i].senders[j].c_str());
        }
        printf("\n");
    }
}

int main() {
    const char* snapshot =
        "CONV 1\n"
        "MSG Alice hi\n"
        "MSG Bob hello\n"
        "CONV 2\n"
        "MSG Charlie sup\n";

    const char* interleaved =
        "CONV 1\n"
        "MSG Alice hi\n"
        "CONV 2\n"
        "MSG Charlie sup\n"
        "MSG Bob hello\n";

    parse_with_back("snapshot", snapshot);
    parse_with_back("interleaved", interleaved);
    return 0;
}
```

  Running this prints:

      snapshot:
        CONV 1 gets: Alice Bob
        CONV 2 gets: Charlie
      interleaved:
        CONV 1 gets: Alice
        CONV 2 gets: Charlie Bob

  Same function, byte for byte. On the snapshot, Bob's message lands with Alice, where it
  belongs. On the interleaved file -- identical lines, one moved -- Bob lands inside Charlie's
  conversation. Nothing in the code changed between the two runs. Only the file's shape did.

  --------
  HOW THIS EXPLANATION COULD HAVE BEEN WRONG, AND HOW TO CHECK
  If the parser had a genuine code bug -- a parsing error, a bad comparison -- it would
  misroute on BOTH files, or at least misroute somewhere on the snapshot too. The prediction
  of the "accidentally correct" explanation is asymmetric and exact: zero errors on any
  snapshot-shaped file, and on an interleaved file, every misrouted message lands specifically
  in whichever conversation's CONV line most recently preceded it -- never in a random one.
  The run above confirms both halves: the snapshot is perfect, and Bob lands exactly where
  back() pointed, in CONV 2, not in some arbitrary wrong place.

  --------
  WHAT I STILL DO NOT KNOW
  How much real 1990s software shipped on this exact bet. Format designers clearly knew the
  danger eventually -- self-describing record formats and length-prefixed framing show up
  everywhere after this era -- but I have not traced whether that arrived as a principle or
  got re-learned one fragmented file at a time, the way it happened here.

  >> STANDARD NAMES
     grouping messages by insertion order, not by ID -> implicit grouping via back()
     a file where physical order matches logical order -> a static snapshot
     the assumption this year's parser depends on but never checks -> one writer, one ordered pass
