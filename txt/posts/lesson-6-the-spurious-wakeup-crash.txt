  SPECIAL -- LESSON 6: THE SPURIOUS WAKEUP CRASH
  Lesson 6: The Spurious Wakeup Crash
  ============================================================================================



  A thread goes to sleep on a condition variable, nobody calls notify, and it wakes up anyway.
  Not a bug in your code -- both POSIX and Windows document this outright: wait can return for
  no reason at all. A "spurious wakeup." Here is what happens if the code trusts that waking
  up means there is work to do.

  --------
  FIRST, FORCE ONE ON PURPOSE
  An experiment fires a random notify_one while the queue is completely empty, on code that
  assumes waking up means data arrived:

```text
    void processQueue() {
        while (true) {
            std::unique_lock<std::mutex> lock(mtx_);

            // waits blindly: no while-loop, no predicate
            cv_.wait(lock);

            // wakes up and assumes there must be data
            std::string msg = log_queue_.front(); // fatal
            log_queue_.pop();
            // ...
        }
    }
```

  --------
  THEN THE RUN, NOT A PREDICTION
  experiment/todo4_spurious_wakeup_bug.cpp crashes instantly with exit code 1: a C++ STL debug
  assertion failure, a segmentation fault.

  Here is the timeline right before it dies:
```text
Starting program...
[Background Thread] Sleeping blindly...
[OS/Kernel] Random Spurious Wakeup sent! (Queue is empty)
[Background Thread] I am awake!
[Background Thread] I am awake, so there must be data! Reading queue.front()...
```
  Process crashes.

  --------
  SO WHY DOES WAKING UP ALONE CRASH IT
  Because cv_.wait(lock) was called with no predicate, the thread just slept. When the OS
  woke it for no reason, it proceeded straight to log_queue_.front() -- no check, no question
  asked about whether there was actually anything there.

  Calling front or pop on an empty queue is undefined behavior. In practice, that is an
  immediate memory violation.

  --------
  SO WAKING UP CAN NEVER BE TRUSTED ON ITS OWN
  A thread waking up must never be treated as proof there is work to do. The condition has to
  be checked directly, either inside a while loop or through wait's own built-in predicate
  form:

```text
// This is exactly what the lambda does internally:
// while (!(stop_ || !log_queue_.empty())) {
//     cv_.wait(lock);
// }

cv_.wait(lock, [this]() { 
    return stop_ || !log_queue_.empty(); 
});
```

  With that predicate in place, a spurious wakeup can no longer reach front or pop:
  1. The OS wakes the thread up for no reason.
  2. wait evaluates the lambda.
  3. The lambda returns false: not stopping, queue still empty.
  4. wait immediately puts the thread back to sleep, on its own, without the code doing
     anything.

  The thread is only allowed past wait if there really is data in the queue, or the server is
  actually shutting down.

  >> STANDARD NAMES
     waking with no notify behind it   -> spurious wakeup
     checking the real condition first -> the predicate form of cv.wait()
     reading/popping an empty queue    -> undefined behavior
