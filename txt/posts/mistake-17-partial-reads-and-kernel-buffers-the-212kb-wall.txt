  SPECIAL -- IPC MISTAKE: PARTIAL READS AND KERNEL BUFFERS THE 212KB WALL
  Mistake 17: Partial Reads and Kernel Buffers The 212KB Wall
  ============================================================================================


  Ask read() for 3 megabytes and it does not wait around collecting 3 megabytes before handing
  control back. It returns whatever is sitting in the kernel's receive buffer right now, which
  on Windows tops out around 212KB, and gives control back immediately -- with no obligation
  to ever come back with the rest on its own.

  --------
  FIRST, WHAT ASKING FOR MORE THAN THE BUFFER HOLDS ACTUALLY GETS YOU
```text
client requests 3MB via read() ---> kernel receive buffer has only ~212KB available
                                             |
                                  returns ~212KB instantly, call already done
                                             |
                       the remaining ~2.8MB requires calling read() again, and again
```
  This mirrors the write side from mistake 7 exactly, from the other end of the same socket:
  a loop checking total_read < requested_bytes is the only way to actually collect everything
  that was sent.

  --------
  THEN THE ACTUAL CODE THIS RUNS ON

```text
#include <stdio.h>
#include <iostream>
#include <cstring>
#include <unistd.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <arpa/inet.h>
#include <signal.h>
#include <netinet/tcp.h>

void support() {
    int server_fd = socket(AF_INET, SOCK_STREAM, 0);
    int opt = 1; 
    setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, (const char*)&opt, sizeof(opt)); 
    
    struct sockaddr_in address;
    address.sin_family = AF_INET;
    address.sin_addr.s_addr = INADDR_ANY;
    address.sin_port = htons(8080);
    
    bind(server_fd, (struct sockaddr*)&address, sizeof(address));
    listen(server_fd, 100);
    
    int client_fd = accept(server_fd, nullptr, nullptr);
    
    char* buffer = new char[100000000]; // 100MB
    memset(buffer, 'A', 100000000);
    
    // ==========================================
    // FILL IN THE CORE LOGIC: SINGLE WRITE
    // No loops! Write 100MB in one go.
    // ==========================================
    int written_now = write(client_fd,  buffer, 100000000);
    sleep(2);
    
    std::cout << "Server successfully pushed " << written_now << " bytes into the network." << std::endl;
    
    close(client_fd);
    close(server_fd);
    delete[] buffer;
}

void consumer() {
    int client_fd = socket(AF_INET, SOCK_STREAM, 0);
    struct sockaddr_in address;
    address.sin_family = AF_INET;
    address.sin_port = htons(8080);
    inet_pton(AF_INET, "127.0.0.1", &address.sin_addr);
    
    connect(client_fd, (struct sockaddr *)&address, sizeof(address));
    
    int requested_bytes = 3000000; // 3MB
    char* buffer = new char[requested_bytes]; 
    
    std::cout << "Consumer asking OS to read exactly 3,000,000 bytes..." << std::endl;
    sleep(2);
    
    // ==========================================
    // FILL IN THE CORE LOGIC: SINGLE READ
    // No loops! Ask for all 3MB in one go.
    // ==========================================
    int bytes_read =  read(client_fd, buffer, requested_bytes)  ;
    
    std::cout << "Consumer actually received: " << bytes_read << " bytes." << std::endl;
    
    close(client_fd);
    delete[] buffer;
}

int main(int argc, char* argv[]) {
    signal(SIGPIPE, SIG_IGN); 
    if (argc < 2) return 1;
    if (strcmp(argv[1], "support") == 0) support();
    if (strcmp(argv[1], "consumer") == 0) consumer();
    return 0;
}
```

  --------
  SO WHY 212KB SPECIFICALLY, AND WHY IT MOVES
  212KB is not a hardcoded constant in this code -- it is whatever the OS's receive buffer for
  this connection happens to be sized at the moment read() is called, and that size is a live
  value the kernel adjusts per connection. The number in this run happens to land near
  212,992 bytes; the full journey post on this blog traces that exact figure back to Windows
  auto-tuning TCP windows per route (RFC 2140), and it can be larger or smaller on a different
  run, a different route, or a different OS entirely.

  What never changes is the contract: read() returns what is currently available, not what was
  asked for, and the only code that reliably collects everything is code written to expect
  that and loop.

  >> STANDARD NAMES
     read() returning less than requested        -> a partial read
     the loop that makes reads reliable            -> loop until total_read == requested_bytes
     the buffer size behind the ~212KB seen here    -> the kernel's receive window for this route
