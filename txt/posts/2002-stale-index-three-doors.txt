  SPECIAL -- 2002: THREE DOORS INTO A READ THAT MISSES ITS OWN DATA
  2002: Three Doors Into a Read That Misses Its Own Data
  ============================================================================================


  Last year's checkpoint named the assumption that was about to break: one writer, one ordered
  pass. This year is the operational change that breaks it, arriving for a reason that has
  nothing to do with correctness -- the database got too big to fit in memory.

  Part of the year-by-year series: [2001](2001-accidental-parser-three-doors.txt) came before
  this; [2003](2003-fragmentation-breach-three-doors.txt) comes next; the whole arc is in
  [The Decade in a Flat File](the-decade-in-a-flat-file.txt).

  --------
  BEFORE ANY DOOR: ONE IDEA, NOTHING ELSE ASSUMED
  A file is a numbered row of bytes, starting at zero. An "offset" is one of those numbers.
  Two stream calls are the whole toolbox here: tellg() asks "which byte am I standing on right
  now?", and seekg(n) jumps the stream straight to byte n without reading anything in between.
  An index, in this post, is nothing grander than a saved list of tellg() answers -- a note
  that says "conversation 1 starts at byte 0" so a later read can seekg(0) instead of scanning
  the whole file. Everything below is about when that note stops being true.

  --------
  FIRST, THE THREE DOORS
```text
door 1: CODEBASE     main() -> build_index() at boot -> print_conversation_fast(id) on read
door 2: HAPPY user    boots, reads an old conversation that has not changed since boot -- instant
door 3: ANGRY user    reads a conversation that just received a live message after boot
```
  Door 3 is, again, not misbehavior. It is the single most ordinary thing a user can do: keep
  talking after the server has been running for a while.

  --------
  WHY THIS YEAR EXISTS AT ALL, FROM DOOR 1
  By 2002 the database is around 5 GB; a contemporary machine has roughly 512 MB of RAM.
  Loading everything into a vector the 2001 way no longer fits. The fix keeps
  only a small index in RAM -- conversation ID mapped to the byte offset where that conversation
  starts on disk -- and seeks directly to that offset on read instead of scanning the whole
  file.

  --------
  ? THE HAPPY PATH, FROM SCRATCH
  Door 2: build_index() walks the file once at boot and records, for every CONV line, the exact
  byte offset where it starts. What single ordering mistake in that walk -- capturing the offset
  at the wrong moment relative to reading the line -- would make every recorded offset point
  into the middle of the wrong line instead of the start of the right one? Name the exact
  moment tellg() has to be called, and why one line off in either direction breaks it.

  --------
  THE ANSWER, TRACED
```text
long current_offset = intercom.tellg();   // must happen BEFORE getline consumes the line
if (!std::getline(intercom, line)) break;
if (line.find("CONV") == 0) { record current_offset for this conv_id; }
```
  tellg() has to run before getline() advances the stream past the line being recorded.
  Capture it after getline() instead, and the offset you record is the position AFTER the CONV
  line -- the start of whatever comes next, not the start of the conversation itself. Every
  seek built from that offset lands one line late for the rest of that entry's life.

  --------
  ?? THE UNHAPPY PATH, FROM SCRATCH -- THE INDEX ITSELF, NOT A TYPO IN BUILDING IT
  Assume build_index() is written correctly, offsets captured at the right moment, index
  entirely accurate at the exact instant boot finished. Door 3: a live message arrives for a
  conversation that was already indexed at boot, and gets appended to the very end of the file,
  the only disk write that skips relocating existing bytes. Walk print_conversation_fast() for that
  same conversation right after: it seeks to the recorded offset and reads forward until it
  hits the next CONV marker. Does it reach the new message? Trace exactly where the read stops
  and why.
```text
byte 0:   CONV 1              <- index says Conversation 1 starts here
byte 7:   MSG Alice Hi!
byte 21:  MSG Bob Hello!
byte 35:  CONV 2              <- reader stops here, sees a new CONV marker
byte 42:  MSG Charlie Sup?
byte 56:  MSG Alice I am back!   <- appended live, AFTER boot, sitting past CONV 2 on disk
```

  --------
  THE ANSWER, TRACED
  The read starts at byte 0, correctly, reads Alice and Bob's messages, then hits byte 35 --
  CONV 2 -- and stops, because the reader's own stopping rule is "read until the next CONV
  marker." Byte 56, Alice's live reply, is never reached. Not because the offset is wrong. The
  offset at byte 0 is completely correct for the file as it existed at boot. The problem is that
  the file described by the index and the file sitting on disk right now are two different
  files, and the index has no way to know that. It was correct the instant it was built and
  wrong the instant the first live append landed anywhere before the conversation's own next-
  CONV boundary.

  --------
  SO THE INVARIANT THIS YEAR ACTUALLY BREAKS
  The index assumes the file is static once built. Live appends are the entire point of running
  a server for more than a few minutes. An index that goes stale on the very first write is not
  a bug in the index code -- it is a structurally incorrect design for a file that keeps
  changing underneath it. Fixing it for real needs one of two things: rebuild the index on every
  append (defeats the purpose of having an index at all), or stop routing by physical position
  entirely, which is exactly what forces the next year's redesign.

  --------
  SEAM. The reasoning ends here; below, the stale index reproduced in C++.

  The full sequence -- build a correct index, append one live message, read through the index,
  watch the message vanish -- in one program. The database is an in-memory stream so the run
  leaves nothing on your disk; tellg() and seekg() behave the same either way, and every offset
  below is printed by the program, not typed in by me:

```cpp
#include <cstdio>
#include <string>
#include <sstream>
#include <vector>

struct IndexEntry {
    int conv_id;
    long offset;
};

std::vector<IndexEntry> build_index(const std::string& db) {
    std::vector<IndexEntry> index;
    std::istringstream in(db);
    std::string line;
    while (true) {
        long offset = (long)in.tellg();
        if (!std::getline(in, line)) break;
        if (line.find("CONV") == 0) {
            IndexEntry e;
            e.conv_id = line[5] - '0';
            e.offset = offset;
            index.push_back(e);
        }
    }
    return index;
}

void read_conv_via_index(const std::string& db, const std::vector<IndexEntry>& index, int target) {
    for (size_t i = 0; i < index.size(); i++) {
        if (index[i].conv_id != target) continue;
        std::istringstream in(db);
        in.seekg(index[i].offset);
        std::string line;
        bool first = true;
        printf("read CONV %d from offset %ld:\n", target, index[i].offset);
        while (std::getline(in, line)) {
            if (!first && line.find("CONV") == 0) break;
            printf("  %s\n", line.c_str());
            first = false;
        }
    }
}

int main() {
    std::string db =
        "CONV 1\n"
        "MSG Alice hi\n"
        "MSG Bob hello\n"
        "CONV 2\n"
        "MSG Charlie sup\n";

    std::vector<IndexEntry> index = build_index(db);
    printf("index built: CONV %d at %ld, CONV %d at %ld\n",
           index[0].conv_id, index[0].offset, index[1].conv_id, index[1].offset);

    long append_at = (long)db.size();
    db += "MSG Alice back again\n";
    printf("appended live message for CONV 1 at offset %ld\n\n", append_at);

    read_conv_via_index(db, index, 1);
    return 0;
}
```

  Running this prints:

      index built: CONV 1 at 0, CONV 2 at 34
      appended live message for CONV 1 at offset 57

      read CONV 1 from offset 0:
        CONV 1
        MSG Alice hi
        MSG Bob hello

  The index was built correctly -- CONV 1 really does start at byte 0, CONV 2 really does start
  at byte 34. The live message really was appended, at byte 57, and the program says so. And
  the indexed read returns three lines and stops. "back again" appears nowhere in the output.
  Nothing errored. The reader hit CONV 2 at byte 34, followed its own stopping rule, and never
  learned byte 57 existed.

  --------
  HOW THIS EXPLANATION COULD HAVE BEEN WRONG, AND HOW TO CHECK
  Two rival explanations are worth killing explicitly. If the append itself had failed, the
  message would not be in the stream at all -- but the program prints the offset it landed at,
  57, past the end of the original 57-byte database, so the bytes are there. If the index were
  simply built wrong -- the tellg()-timing bug from the happy path -- the FIRST read, before any
  append, would already misbehave, starting mid-line. It does not: the read opens cleanly on
  "CONV 1" at byte 0. Both rivals dead, one explanation left standing: correct index, correct
  append, and a stopping rule that makes them incompatible.

  --------
  WHAT I STILL DO NOT KNOW
  Real databases solved this without rebuilding the index on every write, and the standard
  answer is to index the new message's location at the moment of the append -- pay a tiny cost
  per write instead of a full rebuild. What I have not worked out from first principles is the
  crash story: if the process dies AFTER the append lands on disk but BEFORE the index entry
  does, the two disagree again -- which sounds like it needs the append log itself to be the
  authority and the index to be rebuildable from it. That ordering problem is, I suspect, the
  actual seed of the write-ahead log, but I have not yet traced a real WAL implementation to
  confirm the lineage.

  >> STANDARD NAMES
     capturing a byte offset after the line, not before -> the tellg()-timing bug (offset desync)
     mapping conversation ID to a byte offset -> the byte-offset index
     an index correct at build time, wrong after the first live write -> a stale index
