  SPECIAL -- LESSON 2: LESSON 2: THREADS, QUEUES, AND STD::FUNCTION
  Lesson 2: Lesson 2: Threads, Queues, and std::function
  ============================================================================================



  If lambdas are just custom classes, how do standard library components like `std::thread` and `std::queue` store and execute them?

  --------
  STD::THREAD EATS RAW OBJECTS
  When you create a worker thread with a lambda:
```text
auto worker_lambda = [this] { while(true) { /* ... */ } };
std::thread t(worker_lambda);
```
  `std::thread` is a template. It happily accepts your raw `WorkerLambda` object, copies or moves it into its internal thread state, and when the OS spins up the thread, it calls `worker_lambda.operator()()`.

  No wrapper is needed because `std::thread` is designed to bind tightly to whatever exact callable type you give it at compilation time.

  --------
  THE QUEUE AND POLYMORPHISM
  However, when you build a ThreadPool queue:
```text
std::queue<std::function<void()>> tasks;
```
  A queue cannot be a generic template for *every possible lambda type*. A queue is a fixed data structure that expects elements of a fixed size and type.

  Since every lambda is a completely unique, compiler-generated class (e.g., `__WorkerLambda_1` vs `__ServerLambda_2`), they cannot be stored in the same queue directly.

  This is where `std::function` comes in.

  --------
  THE STD::FUNCTION WRAPPER
  `std::function<void()>` is a polymorphic wrapper. It acts as a universal box that can hold *any* callable object (as long as it takes no arguments and returns `void`).

  When the Main Thread enqueues the Server lambda:
```text
tasks.push([this, socket] { handleClient(socket); });
```
  The compiler actually constructs the `ServerLambda` object, and then constructs a `std::function` object that takes ownership of it. The `std::function` object is what physically enters the queue.

  --------
  THE WRAPPER'S OPERATOR()
  When the Worker Thread pulls the task from the queue:
```text
std::function<void()> task = tasks.front();
task();
```
  Because `task` is an object of type `std::function`, the CPU enters `std::function::operator()`.
  Inside this function, the standard library looks at the object it is holding (the `ServerLambda`), and dynamically calls its `operator()`.

  **Key Takeaway**: `std::function` is a box with a button (`operator()`). When you press the button on the box, the box presses the button on whatever object is inside it.
