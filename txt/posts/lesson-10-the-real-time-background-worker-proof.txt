  SPECIAL -- LESSON 10: THE REAL-TIME BACKGROUND WORKER PROOF
  Lesson 10: The Real-Time Background Worker Proof
  ============================================================================================



  Lessons 7 and 8 fixed the predicate. Here is the claim that fix was supposed to make true:
  that processQueue does its work in real time, while main is off doing something else
  entirely, not only at the very end when the destructor runs.

  experiment/todo4_realtime_proof.cpp uses the fully fixed predicate:
```text
cv_.wait(lock, [this]() { return stop_ || !log_queue_.empty(); });
```
  and runs this sequence in main: push "Log 1", sleep a full second, push "Log 2", sleep a
  full second, then let the destructor run.

  --------
  THEN THE RUN, WATCH HOW IT INTERLEAVES

```text
[main] Program started.

[Main Thread] Constructor called! Spawning Background Thread...
[Background Thread] processQueue() started! Entering while(true) loop...

[Main Thread] Pushing to queue: Log 1: User Logged In
[main] Sleeping for 1 full second. Watch the background thread work while I sleep!
[Background Thread] ---> DOING REAL WORK: Wrote 'Log 1: User Logged In' to disk!

[Main Thread] Pushing to queue: Log 2: User Clicked Button
[main] Sleeping for another 1 second...
[Background Thread] ---> DOING REAL WORK: Wrote 'Log 2: User Clicked Button' to disk!

[main] Ending scope. Destructor will NOW be called.
[Main Thread] Destructor called! Setting stop_ = true...
[Background Thread] Queue is empty and stop_ is true. Breaking out of loop!
[Background Thread] processQueue() function has officially ended.
[Main Thread] Destructor finished.
[main] Program ended successfully.
```

  --------
  SO THE WORK HAPPENS WHILE MAIN IS ASLEEP, NOT AT THE END
  Look at what happens right after main pushes "Log 1" and goes to sleep for a full second.
  While main is completely asleep, processQueue wakes up on its own, pulls the log, and writes
  it to disk -- long before the destructor is ever called.

  The thread spawned by the constructor runs fully in parallel with main. With the predicate
  checking the queue, it wakes the instant a log is pushed, writes it, and goes back to sleep,
  keeping RAM empty and the log already safe on disk.

  The destructor turns out to be nothing more than the last knock on the door: we are shutting
  down now, you can go home. All the real work already happened while main was asleep.

  >> STANDARD NAMES
     work done as data arrives, not at exit -> real-time background processing
     the destructor's actual remaining job  -> stop the loop and join, not flush a backlog
