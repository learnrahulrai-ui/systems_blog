  SPECIAL -- LESSON 3: THE FULL EXECUTION CHAIN (END-TO-END)
  Lesson 3: The Full Execution Chain (End-to-End)
  ============================================================================================



  Lesson 1 showed a lambda is just an object. Lesson 2 showed function is a box that forwards
  a call to whatever object it holds. Put both together and trace one real integer,
  client_socket, from the moment the OS hands it over to the moment handleClient actually
  runs on a different thread.

  --------
  FIRST, THE SETUP, BY ADDRESS
  1. The main thread creates the ThreadPool at address 0x5000.
  2. The ThreadPool constructor builds the WorkerLambda, copies 0x5000 into its backpack, and
     hands it to std::thread.
  3. The worker thread spins up and enters WorkerLambda::operator().
  4. The main thread creates the Server at address 0x1000.
  5. The main thread accepts a connection, yielding client_socket = 42.
  6. The main thread builds the ServerLambda, copies 0x1000 and 42 into its backpack.
  7. The main thread wraps the ServerLambda inside a function and drops it in the queue.

  --------
  THEN THE STACK PROVES IT, FRAME BY FRAME
  When the worker thread pulls the function out of the queue and calls it, the call stack does
  not flatten into one function calling another the way it feels like in the source. Each
  layer is a distinct C++ class invoking the next, and each layer carries its own isolated
  this pointer.

  Here is the actual Windows debugger (cdb.exe) stack trace proving that:

```text
--- FRAME 6: WorkerLambda::operator() ---
06 dump_all_this!<lambda_73fe...>::operator()
000000e8`f53ff830 class main::__l2::<lambda_73fe...> * this = 0x00000287`56d08b60

        ↓ (WorkerLambda calls task())

--- FRAME 5: std::function::operator() ---
05 dump_all_this!std::_Func_class<void>::operator()
000000e8`f53ff800 class std::_Func_class<void> * this = 0x00000287`56d08b60

        ↓ (std::function unpacks the stored callable)

--- FRAME 2: ServerLambda::operator() ---
02 dump_all_this!<lambda_a8ad...>::operator()
000000e8`f53ff760 class main::__l2::<lambda_a8ad...> * this = 0x00000287`56d08b68

        ↓ (ServerLambda unpacks its backpack and calls captured_server->handleClient)

--- FRAME 1: Server::handleClient ---
01 dump_all_this!Server::handleClient
000000e8`f53ff730 class Server * this = 0x000000e8`f52ff730
```

  --------
  SO LOOK AT WHAT TYPE EACH THIS ACTUALLY IS
  Read the four this pointers in order:
  1. class main::__l2::<lambda_73fe...> -- the worker lambda
  2. class std::_Func_class<void> -- the function wrapper
  3. class main::__l2::<lambda_a8ad...> -- the server lambda
  4. class Server -- the server

  The raw numeric addresses look close because of how the heap and the local stack scopes
  happened to lay out that run, but the C++ type system keeps every one of those four objects
  separate regardless.

  The worker lambda never turns into the server. It is one object calling a wrapper object
  calling another object, which finally reaches the server object -- four real function calls
  on four real this pointers, not one function magically becoming another.

  >> STANDARD NAMES
     the call chain proven by cdb   -> WorkerLambda -> std::function -> ServerLambda -> Server
     each frame's own this          -> the object that frame's operator() belongs to
