  SPECIAL -- LESSON 8: TOTAL DATA LOSS (THE POWER OUTAGE BUG)
  Lesson 8: Total Data Loss (The Power Outage Bug)
  ============================================================================================



  Lesson 7 already showed the flush-on-shutdown predicate hoards logs in memory until the
  destructor runs. Here is what that strategy is worth the moment the destructor never gets a
  chance to run at all.

  experiment/todo4_power_outage_bug.cpp forces exactly that: a hard crash before shutdown.
  Compile it yourself, no Makefile, no CMake:
```text
g++ -std=c++17 -o demo lesson-8-power-outage-bug.cpp
./demo
```
```text
#include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <chrono>
#include <string>
#include <cstdlib> // for std::abort()

class BadLogger {
public:
    BadLogger() {
        background_writer_ = std::thread(&BadLogger::processQueue, this);
    }

    ~BadLogger() {
        std::cout << "[Logger::~Logger] Shutting down safely...\n";
        {
            std::unique_lock<std::mutex> lock(mtx_);
            stop_ = true;
        }
        cv_.notify_one();
        if (background_writer_.joinable()) background_writer_.join();
    }

    void log(const std::string& msg) {
        {
            std::lock_guard<std::mutex> lock(mtx_);
            log_queue_.push(msg);
        }
        cv_.notify_one();
    }

    void processQueue() {
        while (true) {
            std::unique_lock<std::mutex> lock(mtx_);

            // the bug: we only wake up if the server is shutting down
            cv_.wait(lock, [this]() { return stop_; });

            if (stop_ && log_queue_.empty()) break;

            std::string msg = log_queue_.front();
            log_queue_.pop();
            lock.unlock();

            std::cout << "[Logger::processQueue] Wrote to disk: " << msg << "\n";
        }
    }

private:
    std::mutex mtx_;
    std::condition_variable cv_;
    std::queue<std::string> log_queue_;
    std::thread background_writer_;
    bool stop_ = false;
};

int main() {
    std::cout << "[main] Starting server...\n";
    BadLogger* logger = new BadLogger();

    std::cout << "[main] Server running normally. Users are making transactions...\n";
    for (int i = 1; i <= 5; ++i) {
        std::cout << "[main] -> Logger::log() User purchased item " << i << "\n";
        logger->log("User purchased item " + std::to_string(i));
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }

    std::cout << "[main] The queue currently holds 5 unwritten logs floating in RAM.\n";
    std::this_thread::sleep_for(std::chrono::milliseconds(500));

    std::cout << "\n=======================================================\n";
    std::cout << "[main] DISASTER STRIKES! Power outage! Segfault!\n";
    std::cout << "[main] Server crashing BEFORE Logger::~Logger() can run!\n";
    std::cout << "=======================================================\n\n";

    // Simulate a hard crash. The OS instantly terminates the process.
    std::abort();

    // We will never reach this safe shutdown
    delete logger;
    return 0;
}
```

  --------
  THEN THE RUN, NOT A PREDICTION
  The server accepts 5 critical user transactions. Then the process hits a fatal std::abort.
  The exact exit code is compiler and OS dependent; what is constant is the crash itself.

  Because the program hard-crashed, the destructor never executed. Because the destructor never
  executed, stop_ was never set to true. Because stop_ was never set, processQueue never woke
  up. Because it never woke up, those 5 transactions were never popped or written to disk --
  purged from RAM the instant the OS killed the process.

  Total data loss. In production, that is five purchases with no record anywhere, and the
  items never ship.

  --------
  SO THE WRITE CANNOT WAIT FOR SHUTDOWN AT ALL
  Logs have to hit disk the millisecond they enter the queue, not at some later flush point
  that a crash can erase:

```text
cv_.wait(lock, [this]() { 
    return stop_ || !log_queue_.empty(); 
});
```

  With that clause in place, processQueue pops and writes "User purchased item 1" to the hard
  drive immediately. If the process crashes 500 milliseconds later, it no longer matters -- the
  data already made it to permanent disk before the crash could reach it.

  >> STANDARD NAMES
     data that only exists in the queue/RAM -> at risk of total loss on any hard crash
     writing the instant data is queued     -> the fix that removes that window entirely
