  SPECIAL -- LESSON 9: THE MISSING BREAK CLAUSE CRASH
  Lesson 9: The Missing Break Clause Crash
  ============================================================================================



  --------
  WE SETUP THE TEST CASES BY HAND
  You correctly set up the condition variable to wake up on logs or shutdown:
```text
cv_.wait(lock, [this]() { return stop_ || !log_queue_.empty(); });
```

  But you asked: *"What if I remove the `if (stop_ && log_queue_.empty()) break;` line?"*

  I built `experiment/todo4_missing_break_bug.cpp` to simulate exactly this. The Main Thread runs, does not push any logs, and then cleanly shuts down the server.

  --------
  WE RECORD THE REAL RUNTIME OUTCOME
  When I ran the experiment, the server crashed with a fatal memory violation (Exit Code 1) during its own shutdown sequence!

  Here is the exact function trace of what happened:
```text
[main] Starting program...
[Background Thread] Sleeping...
[Main Thread] Shutting down. Setting stop_ = true and waking thread...
[Background Thread] Woke up! (Queue is empty: YES)
[Background Thread] No break clause! Blindly calling queue.front()...
```
  *Process instantly segfaults and dies.*

  --------
  EXPLANATION (WHY YOU FAILED)
  When `main()` shuts down the server, it calls `~Logger()`, which sets `stop_ = true` and fires `cv_.notify_one()`.

  The Background Thread wakes up. The lambda checks `return stop_ || !log_queue_.empty();`. Since `stop_` is true, the lambda returns true, and the thread is allowed to pass the tollbooth!

  It moves downward. Because you deleted the `break;` statement, it continues straight to:
```text
std::string msg = log_queue_.front(); // FATAL CRASH
```
  The queue is completely empty! Calling `.front()` on an empty queue is Undefined Behavior and crashes the server.

  --------
  WE CORRECT THE SEAM BY LEAVING THE QUEUE DRAINED
  You MUST include the break clause!
```text
if (stop_ && log_queue_.empty()) {
    break; 
}
```
  This is the only way the `while (true)` loop can ever gracefully terminate. When the server shuts down, the background thread wakes up, completely drains the queue to disk, and when the queue is finally empty, it hits this `break` clause, exits the function, and safely dies.
