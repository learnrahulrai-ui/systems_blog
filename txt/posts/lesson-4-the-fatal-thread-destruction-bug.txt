  SPECIAL -- LESSON 4: THE FATAL THREAD DESTRUCTION BUG
  Lesson 4: The Fatal Thread Destruction Bug
  ============================================================================================



  Delete one destructor and a program that ran fine a hundred times in a row crashes on the
  hundred-and-first, every time, at the exact same line -- not inside your logic, but at the
  closing brace of main. Here is the experiment that isolates exactly why.

  --------
  FIRST, THE BUG IN ISOLATION
  A standalone experiment, experiment/todo1_shutdown_bug.cpp, mimics the Logger exactly, minus
  one thing: the destructor never signals its background thread to stop.

  Here is the exact code:
```text
class BadLogger {
public:
    BadLogger() {
        // We start the background thread
        background_writer_ = std::thread(&BadLogger::processQueue, this);
    }

    ~BadLogger() {
        // intentionally missing the shutdown sequence:
        // no stop_ = true, no cv notify, no join
        std::cout << "Destructing BadLogger...\n";
    }

private:
    std::thread background_writer_;
    // ...
};

int main() {
    std::cout << "Starting program...\n";
    {
        BadLogger logger;
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    } // ~BadLogger is called right here!
    std::cout << "Program ended successfully (You will never see this line).\n";
    return 0;
}
```

  --------
  THEN THE RUN, NOT A PREDICTION
  Compiling and running todo1_shutdown_bug.exe crashes the process instantly with exit code 1
  and a Windows C++ runtime error dialog stating: abort() has been called.

  The final std::cout line inside main never executes. The program dies before it can print
  the one line that would prove it exited cleanly.

  --------
  SO WHY DOES DESTROYING THE OBJECT KILL THE PROCESS
  When BadLogger goes out of scope, C++ begins destroying its member variables one by one, and
  one of those members is a std::thread named background_writer_.

  The C++ standard committee wrote one very specific, aggressive rule for threads: if a thread
  object is destroyed while it is still joinable -- still running -- the program must
  immediately call std::terminate and abort. No warning, no exception to catch.

  The background thread was still inside its infinite loop, asleep on cv_.wait, so it was
  still running. The moment the destructor finished and the thread object itself was
  destroyed, that rule fired, and the crash was immediate and unrecoverable.

  --------
  SO THE THREAD HAS TO EXIT BEFORE ITS OBJECT IS DESTROYED
  Fixing this means the background thread has to gracefully leave its loop before the thread
  object is destroyed, not after:

  1. Signal -- set a flag, stop_ = true, so the loop knows it is time to quit.
  2. Wake up -- if the thread is asleep on a condition variable, call cv.notify_one so it wakes,
     checks the flag, and breaks out of the loop.
  3. Join -- call background_writer_.join(), which pauses the destructor until the background
     thread has actually finished executing and returned from processQueue.

  Once those three run in order, the thread object is no longer "running" at the moment it is
  destroyed, and the abort rule never fires.

  >> STANDARD NAMES
     a thread still running at destruction -> joinable
     the standard's hard rule on this      -> std::terminate() on destroying a joinable thread
     the three-step safe shutdown          -> signal, wake, join
