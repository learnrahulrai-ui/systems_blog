  SPECIAL -- LESSON 27: ASSIGNMENTSYSTEM — INDUSTRY & ACADEMIC VALIDATION
  Lesson 27: AssignmentSystem — Industry & Academic Validation
  ============================================================================================



  Every logger lesson so far traced one producer-consumer queue. This one is the same pattern
  wearing a different name: a live chat agent routing engine, built in mini-server/src/
  assignment.cpp and assignment.h, checked line by line against what Intercom and Zendesk
  actually run in production, and against what four university systems courses actually teach.

  --------
  FIRST, THE FIVE FUNCTIONS THAT MAKE IT UP
  assign(conversation_id) -- finds the agent with the lowest active count still below their
  limit. If none are available, pushes to waiting_queue_. Returns the agent's name or
  "QUEUED". assignment.cpp:44-87.

  releaseConversation(agent_name, conversation_id) -- decrements the agent's active count,
  erases the entry from assignments_, pops from waiting_queue_ if anyone is waiting, and
  reassigns to the same agent. assignment.cpp:100-138.

  heartbeat(agent_name) -- finds the agent, writes nowSeconds() into last_heartbeat. A
  proof-of-life signal. assignment.cpp:140-159.

  sweeperLoop() -- a background thread. Sleeps 5 seconds, locks the mutex, scans every agent.
  If now - last_heartbeat exceeds 15 seconds, declares the agent dead, rescues their
  conversations into waiting_queue_, resets active to 0. assignment.cpp:170-219.

  preview(count) -- a stub. Returns the top N agents. assignment.cpp:90-98.

  --------
  THEN THE DATA STRUCTURES THOSE FIVE FUNCTIONS SHARE
```text
std::unordered_map<std::string, Agent> agents_;       // agent name -> Agent struct
std::unordered_map<int, std::string>   assignments_;  // conversation_id -> agent name
std::queue<int>                        waiting_queue_; // FIFO queue of unassigned conversations
std::mutex                             mtx_;          // single lock protecting all shared state
std::atomic<bool>                      sweeper_running_; // stop flag for background thread
int                                    tick_;         // monotonic tiebreaker counter
```
```text
struct Agent {
    std::string name;
    int active;                // how many conversations they currently handle
    int limit;                 // max conversations they can handle
    int last_assigned;         // tiebreaker: lower = assigned longer ago
    long long last_heartbeat;  // seconds since epoch of last heartbeat
};
```

  --------
  THEN A LIVE RUN AGAINST THREE AGENTS, A LIMIT OF TWO EACH
```text
Req 1 (101) -> Assigned to Charlie
Req 2 (102) -> Assigned to Bob
Req 3 (103) -> Assigned to Alice
Req 4 (104) -> Assigned to Charlie    -- all 3 agents now at active=1
Req 5 (105) -> Assigned to Bob        -- all 3 agents now at active=2 (full)
Req 6 (106) -> Assigned to Alice      -- all 3 agents full
Req 7 (107) -> "QUEUED"               -- waiting_queue_ = [107]

Release Alice's conversation 103:
{
    "released": 103,
    "reassigned_from_queue": 107       -- customer 107 rescued from the queue
}
```

  --------
  SO WHERE DOES THIS ACTUALLY DIFFER FROM THE REAL PRODUCTS
  The difference is scale, not logic:
      component              this implementation           industry implementation
      queue storage           std::queue<int> in RAM        Redis, Kafka, or RabbitMQ (survives crash)
      heartbeat storage       long long in RAM               Redis with TTL expiry (automatic cleanup)
      dead agent detection    background thread polling 5s   Redis key expiry event or worker service
      lock mechanism          std::mutex (single process)    Redis atomic transactions (distributed)
      assignment map          std::unordered_map in RAM      PostgreSQL / DynamoDB (persistent)

  --------
  AND THE SAME PATTERN IS THE CORE OF FOUR UNIVERSITY SYSTEMS COURSES

  UC Berkeley, CS162 (Operating Systems) builds a thread pool, a producer-consumer queue, and
  a bounded buffer. The overlap: a mutex protecting shared queue state, worker threads
  consuming from a thread-safe queue, capacity-limited bounded buffers as backpressure, and
  graceful shutdown via atomic flags.

  Vanderbilt's systems programming course, taught by Douglas Schmidt, covers the Active
  Object pattern, the Monitor Object pattern, and heartbeat protocols -- reference text "C++
  Network Programming, Vol. 2". The overlap: heartbeat-based liveness detection, background
  watchdog threads that detect dead workers, thread-safe state management with a mutex and
  condition variables.

  University of Illinois Chicago covers work-stealing task schedulers and load balancers. The
  overlap: capacity-limited task queues, a thread pool with per-worker deques, and
  std::atomic<bool> stop flags for graceful termination.

  University of Southampton covers fault-tolerant concurrent systems. The overlap:
  heartbeat and watchdog monitoring threads, automatic recovery and reassignment when workers
  die, and debounce logic to prevent flapping.

  --------
  SO NAME THE THREE PATTERNS THIS CODE IMPLEMENTS AT ONCE
```text
+----------------------------------------------------------------------+
|  Pattern 1: Producer-Consumer                                        |
|  Producer: assign() pushes to waiting_queue_                         |
|  Consumer: releaseConversation() pops from waiting_queue_            |
|  Textbook: Tanenbaum, Modern Operating Systems, ch. 2                |
+----------------------------------------------------------------------+
|  Pattern 2: Monitor Object                                           |
|  A single mutex (mtx_) protects all shared state                     |
|  Every public method acquires the lock before touching memory        |
|  Textbook: Schmidt, Pattern-Oriented Software Architecture           |
+----------------------------------------------------------------------+
|  Pattern 3: Heartbeat / Failure Detector                             |
|  Writer: heartbeat() records a proof-of-life timestamp               |
|  Reader: sweeperLoop() detects the absence of proof-of-life          |
|  Textbook: Chandra & Toueg, Unreliable Failure Detectors, 1996       |
+----------------------------------------------------------------------+
```

  --------
  SO THIS IS THE THIRD PRODUCER-CONSUMER QUEUE IN THE SAME SERVER
```text
+----------------------------------------------------------------------+
|  Queue 1: thread pool (thread_pool.h)                                |
|  Type:     std::queue<std::function<void()>>                        |
|  Producer: acceptLoop() pushes HTTP sockets                          |
|  Consumer: 16 worker threads pop and handle HTTP                     |
|  Problem:  100 customers/sec, only 16 threads                        |
+----------------------------------------------------------------------+
|  Queue 2: logger (logger.h)                                          |
|  Type:     std::queue<std::string>                                   |
|  Producer: any thread calls Logger::log("message")                   |
|  Consumer: background logger thread flushes to disk                  |
|  Problem:  16 threads log at 1000/sec, disk writes at 100/sec        |
+----------------------------------------------------------------------+
|  Queue 3: assignment (assignment.h)                                  |
|  Type:     std::queue<int>                                           |
|  Producer: assign() pushes conversation_id when all agents are full  |
|  Consumer: releaseConversation() pops when an agent frees up         |
|  Problem:  10 customers/min, only 6 agent slots available            |
+----------------------------------------------------------------------+
```
  All three are the same mathematical shape: producer, queue, consumer.

  --------
  SO WHAT INTERVIEW QUESTIONS DOES THIS CODE ACTUALLY ANSWER
  Why not reject the customer instead of queuing them? Dropping a paying customer is lost
  revenue -- the queue holds them until capacity opens.

  What if the queue grows to 10,000? RAM fills up. The queue needs a max size; once full,
  return "QUEUE_FULL" as backpressure.

  Is the queue persistent? No. std::queue is RAM-only -- a server crash loses every queued
  customer. The fix is writing the queue to Redis or a database instead.

  What is the ordering guarantee? FIFO. std::queue guarantees first-in-first-out, so customer
  107 is served before 108.

  Can releaseConversation deadlock with assign? No. Both acquire exactly one mutex, mtx_.
  Deadlock requires two mutexes taken in different orders, and there is only one here.

  What happens erasing from a map while iterating it? Iterator invalidation. erase() returns
  the next valid iterator, so the loop must capture it: it = map.erase(it), and must not call
  ++it right after an erase.

  Why is the heartbeat separate from the sweeper? Separation of concerns -- the heartbeat is
  the writer, recording time; the sweeper is the reader, checking time. They run on different
  threads at different frequencies.

  What if an agent's network is flaky? Without debounce logic, the sweeper repeatedly kills
  and resurrects the same agent -- flapping. The fix requires N consecutive missed heartbeats,
  not just one.

  --------
  THEN THE TWO ITERATORS INSIDE SWEEPERLOOP, WHICH WALK DIFFERENT MAPS
  Both it and it_in are std::unordered_map::iterator -- a pointer to one key-value entry,
  dereferenced with ->first for the key and ->second for the value. They walk two different
  maps for two different jobs.

  it walks agents_:
```text
agents_  (std::unordered_map<std::string, Agent>)

  key (std::string)    value (Agent struct)
  -----------------------------------------------
  "Alice"             { active=2, limit=2, hb=0 }   <- it (Alice's turn)
  "Bob"               { active=2, limit=2, hb=0 }   <- it (Bob's turn)
  "Charlie"           { active=2, limit=2, hb=0 }   <- it (Charlie's turn)

  it->first  = "Bob"         (agent name, string)
  it->second = Agent struct  (active, limit, last_heartbeat)
```
  it_in walks assignments_:
```text
assignments_  (std::unordered_map<int, std::string>)

  key (int)   value (std::string)
  --------------------------------
   101        "Charlie"
   102        "Bob"          <- it_in (Bob's conversation)
   103        "Alice"
   104        "Charlie"
   105        "Bob"          <- it_in (another of Bob's)
   106        "Alice"

  it_in->first  = 102        (conversation id, integer)
  it_in->second = "Bob"      (agent name, string)
```
  Nested, they answer "which conversations belong to this dead agent":
```text
outer loop: it scans agents_
  it -> "Bob" { hb=0 }
  now(20) - 0 = 20 > 15.  Bob is dead.

    inner loop: it_in scans assignments_ looking for Bob
      it_in -> { 102 -> "Bob" }
        it_in->second == it->first ?
        "Bob"         == "Bob"     ?  yes
          waiting_queue_.push(it_in->first)       -- push 102 (the int)
          it_in = assignments_.erase(it_in)       -- advance safely

      it_in -> { 105 -> "Bob" }
        "Bob" == "Bob"?  yes
          waiting_queue_.push(105)
          it_in = assignments_.erase(it_in)
```
      it              it_in
      map: agents_    map: assignments_
      ->first: string (agent name)   ->first: int (conversation id)
      ->second: Agent struct         ->second: string (agent name)
      job: find dead agents          job: find conversations owned by a dead agent
      loop: for (no erase during scan)   loop: while (erase during scan, catch the return)

  A for loop is safe for it over agents_ because nothing is erased from agents_ during the
  sweeper scan -- ++it always lands on a valid bucket. A while loop is required for it_in over
  assignments_ because entries are erased mid-iteration: erase() invalidates the current
  iterator but returns the next valid one, so the code must capture it as
  it_in = assignments_.erase(it_in), and only call ++it_in on the iterations where nothing was
  erased.

  --------
  SO TRACE WHO ACTUALLY PUSHES INTO WAITING_QUEUE_
```text
waiting_queue_ type: std::queue<int>  -- stores conversation IDs only

producers (push):
  1. assign()       -- when all agents are full
                        waiting_queue_.push(conversation_id)
  2. sweeperLoop()   -- when an agent dies
                        waiting_queue_.push(it_in->first)
                        -- it_in->first is the conversation ID (int)
                        -- it_in->second (agent name) is NOT pushed
                        -- it (the agent struct) is NOT pushed

consumer (pop):
  releaseConversation() -- when an agent finishes a chat
    next_id = waiting_queue_.front()
    waiting_queue_.pop()
```
  And across a full run, both producers actually fire:
```text
t=0    assign(101) -> Charlie     waiting_queue_ = []
t=0    assign(102) -> Bob         waiting_queue_ = []
t=0    assign(103) -> Alice       waiting_queue_ = []
t=0    assign(104) -> Charlie     waiting_queue_ = []
t=0    assign(105) -> Bob         waiting_queue_ = []
t=0    assign(106) -> Alice       waiting_queue_ = []
t=0    assign(107) -> QUEUED      waiting_queue_ = [107]      <- assign() pushed
t=20   sweeper kills all 3        waiting_queue_ = [107, 102, 105, 103, 106, 101, 104]
                                                       ^^^  ^^^^^^^^^^^^^^^^^^^^^^^^^^
                                                       |    sweeperLoop pushed via it_in->first
                                                       assign() pushed earlier
```
