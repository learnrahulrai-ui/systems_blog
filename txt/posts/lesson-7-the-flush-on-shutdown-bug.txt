  SPECIAL -- LESSON 7: THE FLUSH-ON-SHUTDOWN BUG
  Lesson 7: The Flush-on-Shutdown Bug
  ============================================================================================



  Lesson 6 fixed the crash by checking a condition before waking up. Half of that condition
  is missing here on purpose: the wait predicate only checks for shutdown, never for new data.

```text
// only wake up if the server is shutting down
cv_.wait(lock, [this]() { return stop_; });
```

  experiment/todo4_stop_only_bug.cpp runs that exact predicate while the main thread pushes
  normal traffic into the queue and calls notify_one, server running, nothing shutting down.
  Compile it yourself, no Makefile, no CMake:
```text
g++ -std=c++17 -o demo lesson-7-flush-on-shutdown-bug.cpp
./demo
```
```text
#include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <chrono>
#include <string>

class BadLogger {
public:
    BadLogger() {
        background_writer_ = std::thread(&BadLogger::processQueue, this);
    }

    ~BadLogger() {
        std::cout << "[Main Thread] Shutting down server. Setting stop_ = true...\n";
        {
            std::unique_lock<std::mutex> lock(mtx_);
            stop_ = true;
        }
        cv_.notify_one();

        if (background_writer_.joinable()) {
            background_writer_.join();
        }
    }

    void log(const std::string& msg) {
        std::cout << "[Main Thread] User hit endpoint! Logging message: " << msg << "\n";
        {
            std::lock_guard<std::mutex> lock(mtx_);
            log_queue_.push(msg);
        }
        cv_.notify_one();
    }

    void processQueue() {
        while (true) {
            std::unique_lock<std::mutex> lock(mtx_);

            // we only wake up if stop_ == true
            cv_.wait(lock, [this]() { return stop_; });

            if (stop_ && log_queue_.empty()) break;

            std::string msg = log_queue_.front();
            log_queue_.pop();

            lock.unlock();
            std::cout << "[Background Thread] Finally writing to disk: " << msg << "\n";
        }
    }

private:
    std::mutex mtx_;
    std::condition_variable cv_;
    std::queue<std::string> log_queue_;
    std::thread background_writer_;
    bool stop_ = false;
};

int main() {
    std::cout << "Starting program...\n";
    {
        BadLogger logger;

        logger.log("Request 1");
        std::this_thread::sleep_for(std::chrono::milliseconds(200));

        logger.log("Request 2");
        std::this_thread::sleep_for(std::chrono::milliseconds(200));

        logger.log("Request 3");
        std::this_thread::sleep_for(std::chrono::milliseconds(200));

        std::cout << "[Main Thread] Notice how the Background Thread is doing absolutely nothing?\n";
        std::cout << "[Main Thread] The queue is just building up in memory...\n\n";
        std::this_thread::sleep_for(std::chrono::milliseconds(500));

    } // server shuts down here, triggering ~BadLogger()

    std::cout << "Program ended successfully.\n";
    return 0;
}
```

  --------
  THEN THE RUN, NOT A PREDICTION
  Here is the exact output:
```text
Starting program...
[Main Thread] User hit endpoint! Logging message: Request 1
[Main Thread] User hit endpoint! Logging message: Request 2
[Main Thread] User hit endpoint! Logging message: Request 3
[Main Thread] Notice how the Background Thread is doing absolutely nothing?
[Main Thread] The queue is just building up in memory...

[Main Thread] Shutting down server. Setting stop_ = true...
[Background Thread] Finally writing to disk: Request 1
[Background Thread] Finally writing to disk: Request 2
[Background Thread] Finally writing to disk: Request 3
Program ended successfully.
```

  --------
  SO WHY DOES THE QUEUE JUST SIT THERE
  return stop_ tells the background thread exactly one thing: never wake up unless the server
  is literally shutting down. Here is the mechanical timeline that predicate produces:
  1. Main thread pushes "Request 1" into the queue.
  2. Main thread calls cv_.notify_one().
  3. Background thread wakes up and evaluates return stop_.
  4. stop_ is false, so the predicate says no, and the thread goes right back to sleep,
     ignoring the message it was just notified about.
  5. This repeats for hours. log_queue_ grows without bound in memory. Nothing reaches disk.

  Then, at shutdown:
  6. Main thread sets stop_ = true and calls notify_one.
  7. Background thread wakes, evaluates return stop_ (now true), and pops every queued log at
     once, all in one burst, right before the process exits.

  --------
  SO THE PREDICATE NEEDS BOTH CONDITIONS, NOT ONE
  This is an asynchronous logger, not a shutdown-only logger. The background thread has to wake
  up if the server is stopping, or if there is data to process -- either one, not just one of
  them.

```text
cv_.wait(lock, [this]() { 
    return stop_ || !log_queue_.empty(); 
});
```

  Adding that one clause makes the thread wake the moment a new log enters the queue, write it
  to disk, and go back to sleep -- instead of hoarding everything in RAM until shutdown.

  >> STANDARD NAMES
     wait predicate that only checks stop_   -> flush-on-shutdown (the bug)
     wait predicate checking stop_ OR queue  -> the correct asynchronous-logger predicate
