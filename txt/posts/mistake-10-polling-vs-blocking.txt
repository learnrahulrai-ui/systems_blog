  SPECIAL -- IPC MISTAKE: POLLING VS BLOCKING
  Mistake 10: Polling vs Blocking
  ============================================================================================


  A plain file was never built for real-time IPC, and the read side proves it: watching a
  file for new data means calling read() in a loop, over and over, and a loop that never
  sleeps is a loop that burns 100% of a CPU core doing nothing.

  --------
  FIRST, WHAT POLLING COSTS AGAINST WHAT BLOCKING COSTS
```text
consumer, polling:  while(true) -> read() -> returns -1/0 -> loop again immediately -> 100% CPU
consumer, blocking: read() -> no data -> OS puts the thread to sleep                -> 0% CPU
                                            | (data arrives)
                                    OS wakes the thread up
```
  Real IPC primitives -- sockets, pipes -- can block: the OS itself parks the thread and only
  wakes it when data actually shows up. A plain file offers no such call. There is no
  read-and-sleep-until-ready option, only read-and-check-again.

  --------
  THEN THE ACTUAL CODE THIS RUNS ON

```text
#include <stdio.h>
#include <iostream>
#include <cstring>
#include <fstream>
#include <fcntl.h>
#include <unistd.h>

void support() {
    // BLANK: Write "hello consumer" to a file named "chat.txt"

    int fd = open("chat.txt", O_WRONLY | O_CREAT, 0644); 
    if (fd == -1) {
        perror("open failed");
        return;
    }

    char buffer[256]; 
    int len = strlen("hello consumer");
    strcpy(buffer, "hello consumer"); 
    write(fd, buffer, len);


    close(fd);
    
}

void consumer() {
    // BLANK: Open "chat.txt", read the greeting into a buffer, and print it to std::cout!
    
    int fd = open("chat.txt", O_RDONLY);
    if (fd == -1) {
        perror("open failed");
        return;
    }
    
    char buffer[256];
    
    // 2. Read up to 255 bytes (leave room for \0)
    int bytes_read = read(fd, buffer, sizeof(buffer) - 1);
    
    // 1. Error check (chore: youdo it)
    if (bytes_read == -1) {
        perror("read failed");
        close(fd);
        return;
    }
    
    // 3. Check read count / EOF (if bytes_read == 0)
    // 4. Put a \0 in the buffer at index [bytes_read]
    // 5. std::cout the buffer
    
    if (bytes_read > 0) {
        buffer[bytes_read] = '\0';
        std::cout << buffer << std::endl;
    }

    close(fd);    
}

int main(int argc, char* argv[]) {
    if (argc < 2) {
        printf("Usage: ./v1 <support|consumer>\n");
        return 1;
    }
    
    // BLANK 1: Check argv[1] for "support"
    // MISTAKE REMINDER: strcmp returns 0 on match! 'if (strcmp(..))' is false on match.
    if ( strcmp(argv[1], "support") == 0 ) {
            support();
            }

    
    // BLANK 2: Check argv[1] for "consumer"
    // MISTAKE REMINDER: Need '== 0' to run on match!
    if ( strcmp(argv[1] , "consumer") == 0 ) { 
        consumer();
        }
    return 0;
}
```

  --------
  SO WHY A FILE FORCES POLLING AND A SOCKET DOESN'T
  A file descriptor for a socket or a pipe carries a readiness concept the kernel understands
  and can act on: no data means put this thread to sleep, and wake it the instant data lands.
  A plain file descriptor carries no such contract -- read() on a file returns immediately
  whether there is new data or not, because "new data" was never a concept the filesystem was
  built to signal.

  That is why the consumer's only option, checking a plain file, is a while loop calling read()
  as fast as the CPU can execute it -- 100% of a core spent checking a file that changes maybe
  once a second. Moving to sockets in v4 replaces that busy loop with a blocking read() that
  the OS itself controls the wakeup for.

  >> STANDARD NAMES
     a loop that never sleeps, checking repeatedly -> polling
     the OS parking a thread until data is ready    -> blocking
