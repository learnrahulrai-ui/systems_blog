  SPECIAL -- LESSON 13: CASE 1 - THE SHUTDOWN DEADLOCK
  Lesson 13: Case 1 - The Shutdown Deadlock
  ============================================================================================



  --------
  WE SETUP THE TEST CASES BY HAND
  We removed `stop_` from the condition variable lambda, leaving only the queue check:
```text
cv_.wait(lock, [this]() { return !log_queue_.empty(); });
```

  --------
  NORMAL RUNTIME FLOW IN OUR MIND MAP
  Your mental trace of the normal execution was **100% flawless**:
  1. `main()` creates the `Logger`, which passes `this` to spawn a new background thread running `processQueue()`.
  2. A user hits the server. `Logger::log()` gets the lock, pushes to the queue, unlocks, and calls `notify_one()`.
  3. The background thread hears `notify_one()` and wakes up.
  4. It evaluates `!log_queue_.empty()`. Since a log was just pushed, this evaluates to **TRUE**.
  5. It passes the wait, skips the break clause, pops the message, drops the lock, and writes to disk.

  You concluded: *"i do not see the problem"*

  --------
  SHUTDOWN SEQUENCE INTRODUCES A FATAL BLIND SPOT
  Your logic was perfectly sound for a server that runs forever, but you forgot that the program eventually has to close! The bug does not happen when logging; it happens during shutdown when the server is idle.

  Let's trace what happens when `main()` decides to shut down:

  1. The server is completely idle. No users are hitting the website. The `log_queue_` is completely **empty**.
  2. `main()` finishes and calls the destructor `~Logger()`.
  3. `~Logger()` acquires the lock, sets `stop_ = true`, unlocks, and calls `notify_one()` to wake up the background thread.
  4. `~Logger()` then calls `background_writer_.join()`. This means `main()` physically halts and says: *"I will wait right here until the background thread finishes its work and dies."*

  --------
  FATAL FLAW LURKS IN THE DESTRUCTOR
  5. Down in the background, `processQueue()` hears the `notify_one()` and wakes up!
  6. It evaluates your modified lambda: `[this]() { return !log_queue_.empty(); }`
  7. Because the server is idle, the queue is empty. `!log_queue_.empty()` evaluates to **FALSE**.
  8. Because it is FALSE, `cv_.wait()` says: *"False alarm! Go back to sleep!"*
  9. The background thread goes permanently back to sleep. It never reaches the `if(stop_) break;` door.

  --------
  SYSTEM DROWNS IN A FATAL SHUTDOWN DEADLOCK
  * `main()` is permanently stuck in `.join()`, waiting for the background thread to finish.
  * The background thread is permanently stuck in `.wait()`, sleeping forever because no more logs will ever be pushed to wake it up.

  They are stuck waiting on each other forever. The server freezes during shutdown and must be forcefully killed by the OS.

  **The Fix:**
  The lambda must include `stop_ ||`. By adding `stop_`, you give the background thread the ability to "hear" the shutdown signal even when the queue is completely empty.
