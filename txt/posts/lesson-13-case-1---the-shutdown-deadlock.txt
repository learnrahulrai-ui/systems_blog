  SPECIAL -- LESSON 13: CASE 1 - THE SHUTDOWN DEADLOCK
  Lesson 13: Case 1 - The Shutdown Deadlock
  ============================================================================================



  Case 1 from the permutation matrix, traced in full: stop_ is dropped from the wakeup lambda,
  leaving only the queue check.
```text
cv_.wait(lock, [this]() { return !log_queue_.empty(); });
```

  --------
  FIRST, THE NORMAL PATH, WHICH LOOKS COMPLETELY FINE
  1. main() creates the Logger, which passes this to spawn a background thread running
     processQueue.
  2. A user hits the server. log() takes the lock, pushes to the queue, unlocks, calls
     notify_one().
  3. The background thread hears notify_one and wakes up.
  4. It evaluates !log_queue_.empty(). A log was just pushed, so this is true.
  5. It passes the wait, skips the break clause, pops the message, drops the lock, writes to
     disk.

  Traced this far, there is no visible problem -- the logic is sound for a server that runs
  forever.

  --------
  BUT A SERVER EVENTUALLY HAS TO SHUT DOWN, AND THAT IS WHERE IT BREAKS
  The bug does not happen while logging. It happens during shutdown, while the server sits
  idle. Trace what main() does when it decides to shut down:

  1. The server is completely idle. No users are hitting it. log_queue_ is empty.
  2. main() finishes and calls the destructor.
  3. The destructor acquires the lock, sets stop_ = true, unlocks, calls notify_one to wake the
     background thread.
  4. The destructor then calls background_writer_.join(). main() halts here, waiting for the
     background thread to finish and exit.

  --------
  SO THE FLAW IS SITTING INSIDE THE DESTRUCTOR'S OWN WAKEUP CALL
  5. processQueue hears the notify_one and wakes up.
  6. It evaluates the modified lambda: return !log_queue_.empty().
  7. The server is idle, the queue is empty, so this evaluates to false.
  8. wait treats false as a false alarm and puts the thread straight back to sleep.
  9. The background thread never reaches the if(stop_) break door. It is asleep and nothing
     will ever wake it again -- no more logs are coming, and stop_ is never checked without a
     log to trigger the check.

  --------
  SO BOTH THREADS END UP WAITING ON EACH OTHER FOREVER
  main() is stuck in join(), waiting for the background thread to finish. The background
  thread is stuck in wait(), asleep forever, because nothing will ever push another log to
  wake it. Neither side can move. The server freezes during shutdown and has to be killed by
  the OS.

  The fix is the stop_ || the lambda was missing: giving the background thread a way to hear
  the shutdown signal even when the queue has nothing left to wake it up with.

  >> STANDARD NAMES
     each side permanently waiting on the other -> deadlock
     the missing half of the wakeup predicate    -> stop_ || !log_queue_.empty()
